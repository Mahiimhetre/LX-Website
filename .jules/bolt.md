## 2024-05-15 - React Component Re-renders in Playground
**Learning:** Found that large child components (DataTable, ShoppingCart, etc.) in `src/pages/Playground.jsx` were re-rendering unnecessarily whenever parent state (like the active `view`) changed, because handler functions were recreated on every render.
**Action:** Applied `React.memo` to child components and wrapped parent handler functions in `useCallback` (using functional state updates to keep dependency arrays empty). This prevents an estimated 10-20ms of render time per component on unrelated state changes.

## 2024-05-18 - Cron Job Concurrency with Promise.all
**Learning:** Using unbounded `Promise.all` with `.map()` over large arrays for background operations (like dispatching hundreds of reminder emails in `cronJobs.js`) is an anti-pattern. While the asynchronous dispatch is fast, it aggressively spikes memory and database connection pools, and can instantly hit rate limits from third-party email providers. Furthermore, replacing it directly with a sequential `for...of` loop without per-iteration error handling (e.g., `try...catch` inside the loop) introduces a critical functional regression where a single failure aborts the entire batch.
**Action:** Replaced unbounded `Promise.all` mapping in background tasks with sequential `for...of` loops that wrap individual awaited operations in `try...catch`. This stabilizes background memory footprint and prevents partial batch failures without significantly degrading total cron run time.
